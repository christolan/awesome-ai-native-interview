## Improved Answer

AI Agent 中 tool calling 的可靠性问题是构建生产级 Agent 的核心挑战，它和传统 API 错误处理有本质区别——传统 API 的调用方是确定性代码，错误是可枚举、可重现的；而 LLM 生成 tool call 是概率性的，同一输入可能产生不同错误，简单 retry 无法保证收敛。

**常见的 tool calling 失败模式有三类：**

第一，**格式错误**——模型生成的 JSON 结构不合法、缺少闭合括号、或字段名拼错。这源于 LLM 本质上是文本生成器，JSON 只是它"模仿"出来的一种输出格式，没有编译器的语法保证。

第二，**参数不匹配**——类型错误（字符串传给了 integer 字段）、缺少必填参数、值超出允许范围（如枚举外取值）。模型可能"猜"一个参数值，而非严格按照 schema 约束。

第三，**幻觉工具调用**——模型调用了工具列表中根本不存在的工具名，或者对工具用途理解偏差而选错工具。这反映了模型对 tool schema 的理解深度不足。

**系统性应对策略：**

**1. 结构化输出（Constrained Decoding）**——这是最根本的解决方案。OpenAI 的 Structured Outputs / Function Calling 模式、Outlines、Guidance 等工具通过在解码阶段约束 token 采样，保证输出严格符合 JSON Schema。它把"事后校验"变成了"事前保证"，从根上消除格式错误和类型不匹配。

**2. 校验 + 重试闭环**——当无法使用 constrained decoding 时，在 tool call 执行前做 schema 校验（如 JSON Schema Validator），校验失败则将错误信息拼入消息历史重新请求模型。关键技巧：重试 prompt 要包含具体的错误描述（"参数 `limit` 应为整数，你传了字符串 `'10'`"），而非笼统的"格式错误"。这就是 **self-correction** 模式。但要注意设置重试上限（通常 2-3 次），避免无限循环。

**3. Tool 定义设计最佳实践**——很多错误可以通过更好的 tool schema 设计来预防：字段 description 写清楚取值范围和类型；枚举字段用 `enum` 而非自由文本；必填和可选字段区分明确；工具数量控制在合理范围（过多工具会加剧幻觉调用）。

**4. 兜底机制**——对无法恢复的 tool call 错误，Agent 应优雅降级：向用户坦诚工具暂时不可用，或 fallback 到模型的通用知识回答，而非崩溃或静默失败。

**和传统错误处理的本质区别总结：** 传统 API 错误处理是确定性的 if-else 逻辑，retry 有数学意义上的收敛保证；LLM tool call 错误处理是概率性的，retry 的收敛性取决于模型能力，需要结合 constrained decoding（消除不确定性）、self-correction（利用模型的自我修复能力）、和 schema 设计（减少出错概率）三管齐下。
