# How Reasoning Models Think: It's All Tokens

When DeepSeek-R1, OpenAI o1, and Claude's Extended Thinking arrived, they felt like a paradigm shift. Models that "think before they speak," producing invisible reasoning chains that dramatically improve accuracy on math, coding, and logic. The demos were impressive. The explanations were vague. Behind the marketing, though, there's a disarmingly simple truth: reasoning models aren't doing anything fundamentally new. It's all tokens.

Understanding why that's true—and why it matters—is the difference between treating reasoning as magic and treating it as an engineering primitive you can reason about, debug, and build on top of.

## Thinking Tokens Aren't Special

The first thing to internalize: **thinking tokens and output tokens are generated by exactly the same mechanism.** Both come from the same autoregressive process—the model predicts the next token given all previous tokens, feeds it back into the context, and repeats. The forward pass is identical. The weights are identical. There is no second model, no auxiliary module, no "reasoning mode" switch in the neural network.

What makes thinking tokens *different* is only who sees them. In a standard chat, every token the model generates goes straight to the user. In a reasoning model, the model generates a block of "thinking" tokens that are consumed internally—used to shape the next token predictions—but hidden from the final output. Only the tokens after the thinking block reach the user.

This means reasoning isn't a capability added *on top of* the base model. It's the base model's existing capability—its ability to reason step-by-step when prompted with "think aloud"—externalized and made systematic. The model was always capable of intermediate reasoning. What changed is that we gave it a dedicated space to do it, trained it to use that space reliably, and stopped showing the scratch work to the user.

## Why "Think Then Answer" Works

The effectiveness of reasoning models falls directly out of how autoregressive generation works. Every token the model generates is conditioned on every token that came before it. When a standard model tackles a complex math problem, it has to compress multi-step reasoning into a single decision: "what's the next token of the answer?" There's no opportunity to backtrack, verify an intermediate result, or explore an alternative approach. The model either gets the first token right or it doesn't—and every subsequent token compounds the error.

Reasoning models break this compression. By externalizing the reasoning steps as tokens, the model can "see" what it just thought. Each intermediate token narrows the uncertainty for the next one. A chain like "First, isolate x... so 2x = 14... therefore x = 7... now verify: 2(7) + 3 = 17... correct" gives every subsequent token a richer, more precise context than "x = 7" alone ever could.

This isn't a new insight—it's the same principle behind chain-of-thought prompting. What's new is baking it into the model itself through reinforcement learning, so the model *chooses* to reason without being prompted. The mechanism is the same; the autonomy is what changed.

## The Reasoning Switch Lives in the Chat Template

If there's no reasoning switch inside the model, where does it live? In the **chat template layer**—the translation layer between your JSON API request and the raw token stream that feeds into the model.

Every chat API has this layer, even if it's invisible. When you send `{"role": "user", "content": "Hello"}`, the chat template translates it into a token sequence like `<|user|>Hello<|assistant|>`. The model never sees JSON—it sees tokens. The chat template is the intermediary.

Reasoning on/off is just a conditional in this translation layer. When reasoning is enabled, the template injects tokens that signal the model to enter thinking mode. When it's disabled, those tokens are omitted. It's an if-else in the prompt engineering pipeline, not a model parameter. This is why the same model can serve both reasoning and non-reasoning requests—the weights don't change, only the prefix tokens do.

For DeepSeek-R1, the boundary markers are explicit: `<｜end▁of▁thinking｜>` signals the end of thinking and the start of the visible answer. These are special tokens in the model's vocabulary, learned during RL training—the model was rewarded for outputting `  when it finished reasoning and ` ` when it was ready to deliver the final answer. OpenAI's o1 hides these markers at the API level, exposing only a reasoning token count, but the mechanism is identical underneath.

## The Unified View

Once you see it, you can't unsee it: reasoning is tokens, answers are tokens, and the switch between them is tokens. There is no mode flag, no branching logic in the forward pass, no separate inference pipeline. There is only "which token comes next," and the answer depends on what tokens came before.

This unified view has practical implications. It means you can debug reasoning failures by examining the token stream. It means the reasoning budget (maximum thinking tokens) is a generation parameter, not a model capability cap. It means that improvements to base model generation—KV cache optimizations, speculative decoding, faster attention kernels—directly improve reasoning performance too, because it's all the same mechanism.

The models didn't learn a new trick. We learned to give them room to do the trick they always had.

## Key Takeaways

- **Thinking tokens and output tokens are generated by the exact same autoregressive mechanism**—there is no separate model, no auxiliary module, no mode switch inside the neural network.
- **Reasoning works because it externalizes intermediate steps**—each token enriches the context for the next, reducing uncertainty in a way that single-shot answer generation cannot.
- **The reasoning on/off toggle lives in the chat template layer**, not the model—it's a prompt engineering primitive that injects or omits special tokens in the prefix.
- **Boundary markers like DeepSeek-R1's `<｜end▁of▁thinking｜>` and ` ` are special tokens learned through RL**, functionally identical to EOS tokens—the model learns *when* to emit them, not *how* they work.
- **Everything is tokens**: reasoning, answers, switches, and boundaries—no branching, no mode flags, just different starting points in the same autoregressive process.
