# How to Design Streaming AI Chat on the Frontend

## Interview Question
你要为一个 Web 应用接入“流式 AI 对话”能力。用户发送问题后，前端需要像 ChatGPT 一样边生成边展示，并且支持：实时展示 token 流、用户可中途停止回答、一轮对话结束后可靠落盘、页面刷新后还能恢复这轮会话。请从前端工程视角说明：你会如何设计交互流程、状态组织、协议选型，以及如何处理停止生成、落盘一致性和网络异常恢复？

## Why this question matters
这道题能快速区分候选人是否真正理解 **AI Native 前端工程的复杂度不在“把文本流出来”，而在状态一致性、控制权划分和异常恢复**。

如果回答者只会说“前端用流式渲染、用户可以点停止”，通常说明还停留在表层实现；更成熟的回答应该能把 **消息状态机、协议取舍、幂等持久化、恢复策略和前后端职责边界** 讲成一套完整方案。

## Reference Answer
我会把这套能力当成一个 **带状态机的对话系统** 来设计，而不是一个简单的“发请求 + 拼字符串”功能。

用户发送消息后，前端先立即插入一条 `user message`，同时创建一条空的 `assistant message` 作为占位，并为这次生成分配 `requestId`。整个会话使用 `conversationId` 标识，每条消息有自己的 `messageId`。随后前端发起流式请求，把本轮上下文和这些标识传给服务端；前端边接收流式 chunk，边把内容 append 到当前 assistant message 上，UI 持续增量渲染。

在协议选择上，如果这是典型的 **服务端单向推送模型输出**，我会优先考虑 **SSE** 或 **fetch + ReadableStream**。因为大模型文本生成本质上是单向流输出，这两种方式更贴合 HTTP 语义，实现复杂度也更低。WebSocket 更适合双向高频实时交互，例如协作编辑、语音流、多人在线状态同步；如果只是做文本流式回答，往往会显得更重。

前端状态我会至少拆成三层：

- **消息状态**：`streaming / completed / stopped / failed`
- **请求状态**：`idle / requesting / aborting / reconnecting`
- **持久化状态**：`not_started / persisting / persisted / persist_failed`

这样可以避免“回答已经显示完了，但数据库还没写成功”这种状态混淆。

“停止生成”是最容易翻车的点之一。我会这样处理：前端点击停止时，先通过 `AbortController` 立即中止本地流读取，让 UI 立刻停下来；同时向服务端发送 `abort(requestId)`。服务端停止模型生成，并以 **服务端已经累计到的输出内容** 作为最终保存依据，然后返回这条 assistant message 的最终状态和持久化结果。这里必须明确：**服务端是最终事实源，前端不是**。因为前端可能丢 chunk、页面可能刷新、网络可能中断，如果由前端以“我看到的内容”为准回传保存，会很容易造成展示内容和数据库内容不一致。

关于“落盘一致性”，服务端应基于 `requestId` 或 `messageId` 做幂等写入。也就是说，同一轮生成无论因为重试、断线重连还是重复提交触发多少次保存动作，最终都只更新同一条 assistant message，而不是插入多条记录。这样可以避免页面刷新后出现重复消息。

页面刷新和网络异常恢复我会分两类：

- **恢复已完成历史**：页面加载时根据 `conversationId` 拉取服务端历史消息，恢复完整会话
- **恢复进行中的生成**：如果服务端还记录该消息处于 `streaming` 或 `aborting` 中间态，前端要么按 `requestId` 重新订阅流，要么把消息标记为“上次生成中断，可重新生成”，避免留下无状态的半截回答

所以这道题的关键不是“把字流出来”，而是设计一套 **前后端状态一致、控制权明确、异常可恢复** 的流式对话系统。

## Follow-up Questions
- SSE、fetch stream 和 WebSocket 在 AI 文本生成场景中的边界分别是什么？
- 如果服务端不支持续流，页面刷新后你会如何设计“半截回答”的 UX？
- “生成完成”和“已落盘成功”为什么必须拆成两个状态？

## Notes
- 好回答要把消息状态、请求状态、持久化状态拆开讲
- 要明确 `conversationId / messageId / requestId` 的职责，而不是笼统说“记录一个 id”
- 进阶回答会强调：停止生成时服务端才是最终事实源，持久化要做幂等
